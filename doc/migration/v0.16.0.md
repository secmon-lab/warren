# Migration Guide: v0.16.0

This guide covers the data migration steps required when upgrading to Warren v0.16.0, which ships the **chat-session-redesign** initiative (Phase 0 – Phase 7).

## Overview

v0.16.0 redefines how Warren models conversations:

- **Session** is now keyed deterministically per Slack thread (`slack_<hash>`) and carries a `Source` (`slack` / `web` / `cli`), a `ChannelRef`, an embedded `Lock`, and `TicketIDPtr`.
- **Turn** is introduced as a new entity (one per `@warren` mention or Web HITL request) that groups related Messages.
- **Message** absorbs the role previously played by `ticket.Comment`. User inputs (Slack mentions, Slack non-mentions, Web typed messages, CLI prompts) and every AI output (response / trace / plan / warning) are persisted as `session.Message` rows.
- **gollem History** is scoped per-Session under `sessions/{session_id}/history.json` instead of per-Ticket `latest/{ticket_id}.json`.
- **GraphQL**: `ticketComments` query and `Ticket.comments` field are removed. Use `ticketSessionMessages(...)` and `Ticket.sessionMessages(...)` instead.
- **Base tool**: `warren_get_ticket_comments` is removed. Use `warren_get_ticket_session_messages` instead.
- **WebSocket**: a unified envelope (`EventKind`, `MessageView`, `SessionView`, `TurnView`) replaces the legacy `message` / `trace` / `warning` / `status` formats. The old shapes are still translated client-side via a shim in `useWebSocket`.

Pre-existing alert / triage / enrichment policies are **unaffected**. The migration concerns Firestore data (Sessions, ticket_comments) and GCS history snapshots only.

## Breaking Changes Summary

| Area | Removed / Changed | Replacement |
|------|-------------------|-------------|
| GraphQL | `ticketComments(ticketID)` query, `Ticket.comments` field, `Comment` type | `ticketSessionMessages(ticketID, source, type, limit, offset)`, `Ticket.sessionMessages(source, limit, offset)`, `SessionMessage` type |
| Base tool | `warren_get_ticket_comments` | `warren_get_ticket_session_messages` (accepts `source`, `type`, `limit`, `offset`) |
| Firestore: `sessions` | `query`, `slack_url`, `status`, `request_id`, `updated_at` are deprecated (read-tolerated, never written) | `Source`, `TicketIDPtr`, `ChannelRef`, embedded `Lock` |
| Firestore: `ticket_comments` | Subcollection deprecated; new writes go to `session.Message` instead | `session.Message(type=user, source=slack)` rows |
| GCS | `latest/{ticket_id}.json`, `history/{ticket_id}/*.json` | `sessions/{session_id}/history.json` |
| WebSocket events | `message` / `trace` / `warning` / `status` (still shimmed at client) | Unified `Envelope { kind, message?, session?, turn? }` |

Custom GraphQL clients that query `ticketComments` or read `Ticket.comments` MUST be updated before deploying v0.16.0. Custom AI policies that reference `warren_get_ticket_comments` MUST be repointed to `warren_get_ticket_session_messages`.

## Recommended Migration Order

The redesign was engineered to be deployed first and migrated second. The new binary auto-heals legacy Sessions at read time (see `pkg/usecase/chat` Source inference), so the application stays functional even before any migration job runs. Migration jobs make the data store consistent with the new schema.

**All five migration jobs are non-destructive.** No job in v0.16.0 deletes pre-redesign Firestore documents or Cloud Storage objects. Legacy `ticket.Comment` rows and legacy `latest/{ticket_id}.json` objects remain in place after the migration; operators can decide separately, outside the scope of this upgrade, when (if ever) to discard them.

1. **Deploy v0.16.0.** Verify Slack mentions, Web Conversation tab, and CLI chat all work against legacy data through read-time auto-heal.
2. **`session-source-backfill`** — assign `Source=slack` and `TicketIDPtr` to legacy Session rows.
3. **`session-consolidate`** — ensure every (Ticket, Slack thread) pair has its canonical `slack_<hash>` Session persisted; legacy UUID Sessions are left in place but filtered out at the UI layer.
4. **`comment-to-message`** — copy each `ticket_comments` row into a `session.Message(type=user, source=slack)` of the corresponding Slack Session.
5. **`turn-synthesis`** — synthesize Turn entities from legacy Sessions and stamp `TurnID` onto Messages.
6. **`history-scope`** — server-side copy of `latest/{ticket_id}.json` into `sessions/{session_id}/history.json`.

Every job is **idempotent** and accepts `--dry-run`. Always run `--dry-run` first; the printed `Scanned / Migrated / Skipped / Errors` counters are how you decide it is safe to apply.

## Migration Steps

### Step 0: Pre-flight

```bash
# A Firestore PITR boundary or export is the cheapest insurance for any
# operator mistake (wrong project, partial interruption, etc.). The
# jobs below only write new data, so a snapshot is not strictly
# required to recover, but it is recommended.
gcloud firestore export gs://${BACKUP_BUCKET}/warren-v0.16.0-pre/
```

### Step 1: Deploy v0.16.0 binary

Roll out the new binary as you normally would. Confirm:

- Slack mentions reach the new ChatFromSlack pipeline (logs: `chat.session.resolve` followed by `chat.activity.lock.acquired`).
- Web Conversation tab loads on a Ticket page and shows historical Slack messages.
- WebSocket clients receive both legacy event kinds and the new envelope (the shim in `frontend/src/hooks/useWebSocket.ts` handles the conversion).

If any of the above fails, roll back the binary. No data has changed yet.

### Step 2: List and dry-run jobs

```bash
warren migrate --list

warren migrate --dry-run \
  --job session-source-backfill \
  --job session-consolidate \
  --job comment-to-message \
  --job turn-synthesis \
  --job history-scope
```

Check the per-job `Scanned / Migrated / Skipped` counts. Errors must be 0 before proceeding.

### Step 3: Apply data migrations

Run them one at a time so log inspection is easy.

```bash
warren migrate --job session-source-backfill
warren migrate --job session-consolidate
warren migrate --job comment-to-message
warren migrate --job turn-synthesis
warren migrate --job history-scope
```

`history-scope` requires `--storage-bucket` to be configured the same way as the running server.

For convenience, the same five jobs are also exposed as a single bundle job named `v0.16.0`:

```bash
warren migrate --dry-run --job v0.16.0
warren migrate --job v0.16.0
```

The bundle invokes the underlying jobs in the same order shown above. Running the bundle is equivalent to running the five jobs individually; each underlying job remains available on its own for partial reruns or focused debugging.

### Step 4: Verify

Let production run for at least a few days against the migrated data. Keep an eye on:

- AI prompt outputs that previously read `ticket.Comment` — they now read `session.Message(type=user)`. Quality regressions usually surface as "the agent forgot what the user said".
- Slack avatar / display name rendering in the Conversation pane (auto-resolved at read time).
- Any custom dashboards or exports that touched `ticket_comments`.

## Idempotency Notes

- `session-source-backfill`: Skipped if `Source` is already valid and `TicketIDPtr` is set.
- `session-consolidate`: Calls `ResolveSlackSession`, which is itself idempotent — re-runs return the existing canonical row.
- `comment-to-message`: De-duplicates by `(content, createdAt, author)` per Session; re-runs detect and skip.
- `turn-synthesis`: Skips Sessions that already have at least one Turn.
- `history-scope`: Skips destination `sessions/{session_id}/history.json` paths that already exist.

You can stop and resume any job at any time. Failed individual rows are reported via the `Errors` counter and never abort the run.

## Rollback Strategy

The key question is: **after running migration jobs, can the original (pre-v0.16.0) data still be read by v0.15.x?**

**Yes.** All five migration jobs are non-destructive and only add new data. v0.15.x's read paths (`ticket_comments`, `latest/{ticket_id}.json`, legacy `Session` fields) are untouched.

### Effect of each job on pre-existing data

| Job | Touches `ticket_comments`? | Touches legacy `latest.json`? | Touches legacy Session fields? |
|-----|----------------------------|-------------------------------|---------------------------------|
| `session-source-backfill` | No | No | Adds `Source` / `TicketIDPtr` / `ChannelRef`. Existing `query` / `slack_url` / `status` / `request_id` / `updated_at` are preserved verbatim by `PutSession`. |
| `session-consolidate` | No | No | No. Only writes brand-new canonical `slack_<hash>` Session rows; legacy UUID Sessions are untouched. |
| `comment-to-message` | Read only | No | No. Writes new `session.Message` rows; the source `ticket.Comment` rows stay exactly as they were. |
| `turn-synthesis` | No | No | No. Creates new Turn entities and stamps `TurnID` onto the new `session.Message` rows; `ticket.Comment` is not read or written. |
| `history-scope` | No | Read only | No. Copies into the new path; the legacy file stays in place. |

So after every migration step, every byte of pre-existing data is still there. The new fields, new rows, and new files are pure additions that v0.15.x ignores.

### Rolling back to v0.15.x

Safe with respect to **original** data:

- v0.15.x reads `ticket_comments` and `latest/{ticket_id}.json` — both are intact.
- New Session fields (`Source`, `TicketIDPtr`, `ChannelRef`, `Lock`) are extra columns that v0.15.x silently ignores; Firestore tolerates unknown fields.
- The canonical `slack_<hash>` Sessions written by `session-consolidate` are unreferenced from v0.15.x's perspective and remain as harmless orphan rows.
- New `session.Message` rows, Turns, and `sessions/{sid}/history.json` files likewise are not read by v0.15.x.

To roll back the binary:

1. Redeploy v0.15.x.
2. Roll back the frontend bundle as well — the v0.16.0 frontend depends on `ticketSessionMessages` / `Ticket.sessionMessages` and the new WebSocket envelope, none of which v0.15.x serves.
3. Custom AI policies that were repointed to `warren_get_ticket_session_messages` must be reverted to `warren_get_ticket_comments`.

The migration artifacts (new fields, Turns, mirrored Messages, copied history files) can simply be left in place. They cost storage but cause no functional issues, and a subsequent re-upgrade to v0.16.0 will pick up where it left off thanks to idempotency.

### What about activity that happens on v0.16.0 itself?

**Chat / comments produced while v0.16.0 was running are not visible to v0.15.x after a rollback.** v0.16.0 stops writing to `ticket.Comment` and `latest/{ticket_id}.json` entirely; user input and AI output are persisted only as `session.Message` rows and as `sessions/{sid}/history.json` files. After a rollback those rows still exist in Firestore but the v0.15.x UI cannot render them. Plan the migration window accordingly.

### Rollback for individual migration jobs

There is no automated "undo" for any individual job. The intended recovery path is: do nothing. The jobs only add data; "bad" runs are recovered by fixing the bug and re-running (each is idempotent).

## Troubleshooting

### `comment-to-message` reports many `Errors`

Most common cause: the legacy Comment lacks `SlackMessageID`, so the job cannot resolve a destination Session. Inspect logs for the failing Comment IDs; they are usually pre-v0.10.0 rows that were already orphaned. They can be safely left in place — they will not be migrated, but nothing in v0.16.0 reads them either.

### Conversation pane shows duplicated Slack messages after migration

The Slack pipeline persists user input as `session.Message` at ChatFromSlack time, and `comment-to-message` mirrors the legacy `ticket.Comment`. If a Comment was created on v0.16.0 (by a code path that was missed during the legacy confinement), both rows can co-exist. The fix is in `pkg/usecase/chat`'s dedup-by-`SlackMessageID` logic; rerunning `comment-to-message` after the fix is a no-op (the dedup guard prevents double-creation).

### `history-scope` skips every Session

This means `sessions/{session_id}/history.json` already exists. That is the idempotency guard; it is not an error. Confirm by running with `--dry-run` and inspecting the `Skipped` counter.

### `session-consolidate` reports `created` for thousands of Sessions on a re-run

This indicates that the canonical-row write was not durable on the previous run (transient Firestore error mid-job). Re-running is the correct response; it will converge.
